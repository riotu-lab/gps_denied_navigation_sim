Overview

gps_denied_navigation_sim is an open-source simulation package that bundles everything needed to develop, tune, and benchmark GPS-denied navigation on multirotor UAVs: a PX4 SITL flight stack, Gazebo Garden worlds with real-terrain DEMs, X500 airframes with different sensor payloads (mono / stereo / twin-stereo, 3D LiDAR, Velodyne), MAVROS plumbing, pre-baked RViz layouts, and turnkey integrations with a dozen estimators.

The package is designed so that a single flight can be compared head-to-head across algorithms: ground truth is always published on /target/gt_path, every launch file spawns the same MAVROS namespace, and three analysis tools — path_error_calculator, data_sync_recorder, and analyze_tercom_log.py — turn live runs into CSVs and publication-quality figures.

Default stack: TERCOM + ESKF terrain-aided navigation on the taif_test4 heightmap world, with MAVROS ground truth, live RViz diagnostics, and CSV logging.
Drop-in estimators: TERCOM, MINS (mono / stereo / twin-stereo+LiDAR), OpenVINS, FAST-LIO, FAST-LIVO2, ORB-SLAM3, RTAB-Map, SPARK-FAST-LIO, KISS-Matcher, RESPLE, SUPER.

UAV Models

Three X500-derived platforms share the same flight controller and MAVROS namespace, differing only in payload.

x500_mono_cam_3d_lidar

Mono camera + 3D LiDAR + gimbal + rangefinder. Default for TERCOM and monocular VIO.

x500_stereo_cam_3d_lidar

Stereo pair + 3D LiDAR. Recommended for MINS-stereo, OpenVINS-stereo, ORB-SLAM3, RTAB-Map.

x500_twin_stereo_twin_velodyne

Four cameras (front + rear stereo) and two Velodyne VLP-16 LiDARs for full 360° perception.

Simulation Worlds

Six Gazebo worlds — one indoor warehouse and five outdoor DEMs derived from SRTM and Mapbox Terrain.

taif_test4 (default)

Heightmap DEM of Taif, Saudi Arabia. Default world for TERCOM.

taif_world

Collada-mesh DEM of the Taif region — larger extent, lower resolution.

taif1_world

Alternative Taif DEM tile (taif1_dem).

taif_test

Single-tile sandbox for heightmap placement debugging.

dem_world

Generic SRTM DEM over the Pacific Northwest for cross-geography tests.

tugbot_depot

Indoor warehouse (30 × 15 × 9 m) with structured obstacles.

TERCOM — default GPS-denied stack

Terrain Contour Matching (TERCOM) estimates the UAV position by correlating a live elevation profile (baro + rangefinder, corrected for attitude) against a pre-loaded GeoTIFF DEM. A 15-state Error-State Kalman Filter fuses the TERCOM fixes with IMU prediction and optional velocity and barometric updates. GPS is used only once at startup to initialise the filter.

TERCOM match lifecycle

sequenceDiagram participant S as Sensors participant C as ProfileCollector participant M as Vectorised Matcher participant E as eskf_node S->>C: h_baro, h_agl, pos_enu, t Note over C: wait until motion creates spacing > min_spacing C-->>C: h_terrain = h_baro - h_agl cos(roll) cos(pitch) alt profile full (N samples) C->>M: sample array + predicted UTM M-->>M: M×N search grid, mask NoData, MAD scores M->>E: position_fix (if accepted) M->>E: match_quality else rejected M-->>E: rejection_reason end

ESKF state machine

stateDiagram-v2 [*] --> WAITING_GPS WAITING_GPS --> INITIALIZING : GPS fix INITIALIZING --> RUNNING : N samples averaged INITIALIZING --> WAITING_GPS : timeout RUNNING --> DIVERGED : NIS / covariance blow-up DIVERGED --> RESETTING : divergence_action RESETTING --> RUNNING : soft reset RESETTING --> WAITING_GPS : hard reset

Four dockable RViz panels (from tercom_rviz_plugins) provide live diagnostics:

  • Filter Status — ESKF state machine, NIS sparkline, σ / innovation gauges, IMU biases
  • TERCOM Quality — pipeline state, MAD / discrimination / roughness / noise bars
  • Error History — live horizontal + vertical error vs. ground truth with RMS / max overlays
  • Profiling — per-callback exec time and call frequency for every tercom_nav node

Benchmark Workflow

Record once, compare many — the simulator is designed around the following loop.

graph LR A[Start sim] --> B[Record bag
GT + raw sensors] B --> C[Replay bag --clock] C --> D[Algorithm 1
→ CSV] C --> E[Algorithm 2
→ CSV] C --> F[Algorithm 3
→ CSV] D --> G[analyze_path_errors.py] E --> G F --> G G --> H[Merged figures
RMSE, max, 3σ consistency]

Three complementary tools:

  • path_error_calculator + run_path_error_analysis.py — online Euclidean / angular / velocity error between any pair of nav_msgs/Path topics with CSV export.
  • data_sync_recorder — builds a replayable dataset with time-synchronised images + IMU + GPS + LiDAR + baro.
  • analyze_tercom_log.py (from tercom_nav) — regenerates 15 publication-quality figures from a single TERCOM CSV (trajectory, RMSE, covariance consistency, NIS, filter health...).

Full walk-through in docs/ALGORITHM_ANALYSIS.md.

Get Started

Clone the simulator and run the installer (this also pulls PX4, MAVROS, mavlink, tercom_nav and tercom_rviz_plugins):

cd ~/shared_volume/ros2_ws/src
git clone https://github.com/riotu-lab/gps_denied_navigation_sim.git
cd gps_denied_navigation_sim
export DEV_DIR=~/shared_volume
./install.sh

Run the default TERCOM stack on taif_test4:

# Terminal 1 — RMW daemon
zenoh

# Terminal 2 — PX4 SITL + Gazebo + MAVROS + RViz
mono_taif4

# Terminal 3 — TERCOM + ESKF + diagnostics
tercom

Or try any other combination:

stereo_taif4   # stereo UAV on taif_test4
twin_taif      # twin-stereo + twin-Velodyne on taif_world
mono_tug       # mono UAV in the tugbot_depot warehouse

Full documentation: docs/ — see UAV_MODEL.md, SIMULATION_ENVIRONMENT.md, ALGORITHMS.md, TERCOM.md, ARCHITECTURE.md, and ALGORITHM_ANALYSIS.md.
